Building linux-dm250 for the Pomera DM250 on an Apple Silicon Mac

Before you start tinkering

Target summary

ItemValue

Mainline does not ship a pomera_defconfig. multi_v7_defconfig is the canonical ARMv7 multi-platform config and already enables CONFIG_ARCH_ROCKCHIP, the RK3128 clock/pinctrl drivers, Lima GPU, etc. Several DM250-specific options are missing and must be added by hand — see config gaps.

Back up stock firmware before doing anything destructive

Do this first. You will lose the proprietary brcmfmac Wi-Fi firmware, the BT patchram, calibration NVRAM, and possibly bootloader env if you reflash blindly. Mainline Linux does not ship these blobs.

What you're trying to save

  • Full eMMC image — one big .img file containing every partition. This is the master backup; everything else can be extracted from it.
  • Wi-Fi/BT firmware blobs — live inside the stock rootfs at paths like /lib/firmware/brcm/brcmfmac43430-sdio.bin, brcmfmac43430-sdio.txt (per-board NVRAM), brcmfmac43430-sdio.clm_blob, plus BT patchram (BCM43430A1.hcd or similar). Without these, brcmfmac loads but the radio stays dark.
  • MAC address / calibration NVRAM — small partition holding per-device data. Captured automatically as part of the full eMMC image.
  • Bootloader (idbloader + u-boot.itb) — captured automatically too.

Recommended method: MASKROM mode + rkdeveloptool over USB

The cleanest backup path. Plug DM250 into MacBook, put the SoC into MASKROM mode, dump the eMMC over USB. No need to boot the stock OS, no need for serial console.

How it works: Rockchip SoCs have a hardware-level recovery state called MASKROM mode. The BootROM enumerates as a USB device. The host pushes a small loader stub into SRAM, which brings up DDR + eMMC, then accepts read/write commands. The stock firmware is bypassed entirely.

1. Install rkdeveloptool on the Mac

No reliable Homebrew formula. Build from source:

brew install autoconf automake libusb pkg-config

git clone https://github.com/rockchip-linux/rkdeveloptool.git
cd rkdeveloptool
autoreconf -i
./configure
make
sudo cp rkdeveloptool /usr/local/bin/
2. Fetch the RK3128 loader blob

The DDR-init + eMMC-bringup loader for RK3128. Rockchip publishes these in rkbin:

git clone https://github.com/rockchip-linux/rkbin.git
ls rkbin/bin/rk31/ | grep -i loader
# e.g. rk3128_loader_v1.13.116.bin

Note the exact filename; you'll pass it to rkdeveloptool db below.

3. Put DM250 into MASKROM mode

Three possible entry methods (try in order):

  1. Key combo at power-on. Hold a specific key (often the recovery/menu key) while pressing power, with USB already connected. Check asciilifeform's DM250 notes for the exact combo — it isn't documented in this fork.
  2. Loader mode via u-boot. If you have serial console, interrupt at the u-boot prompt and run rockusb 0 mmc 0 — u-boot then exposes the eMMC as a Rockchip USB device. Same protocol, slightly different mode (Loader mode rather than MASKROM).
  3. Short the MASKROM test point. RK3128 has a test point on the PCB that, when shorted to ground at power-on, forces MASKROM regardless of eMMC contents. Last-resort method — requires opening the case and identifying the pad. Photo references for DM250 may exist in asciilifeform's writeup.
4. Confirm the Mac sees the device
rkdeveloptool ld
# Expected output (MASKROM mode):
# DevNo=1   Vid=0x2207,Pid=0x310b,LocationID=...    Maskrom
#
# Or in Loader mode:
# DevNo=1   Vid=0x2207,Pid=0x310b,LocationID=...    Loader
Vid 0x2207 is Rockchip's USB vendor ID. Pid 0x310b identifies RK3128 family. If rkdeveloptool ld prints nothing, the device is not in MASKROM/Loader mode — recheck the key combo. system_profiler SPUSBDataType | grep -i rockchip on the Mac is a useful sanity check.
5. Push the loader (MASKROM mode only)

Skip this step if rkdeveloptool ld showed "Loader" instead of "Maskrom".

rkdeveloptool db rkbin/bin/rk31/rk3128_loader_v1.13.116.bin
# "Downloading bootloader succeeded."

After this, the chip is ready to read/write eMMC.

6. Inspect what's on eMMC
# Read flash info: size, sector count, page size
rkdeveloptool rfi

# Read the partition table (GPT or Rockchip parameter file)
rkdeveloptool ppt

Note the total sector count from rfi — you'll need it for the full dump.

7. Dump the entire eMMC
# Replace <SECTORS> with the total sector count from `rfi`.
# Each sector is 512 bytes. A 4 GB eMMC ≈ 7,633,920 sectors.
rkdeveloptool rl 0 <SECTORS> dm250-stock.img

# Compress for storage:
zstd -19 dm250-stock.img       # ~5x smaller than raw
sha256sum dm250-stock.img > dm250-stock.img.sha256
8. Reboot the device back to normal
rkdeveloptool rd        # Reset device, power back to stock

Extracting the Wi-Fi/BT blobs from the dump

Once you have dm250-stock.img, mount its partitions to extract /lib/firmware/brcm/. Easiest in Linux (do it in the same Docker builder container):

# Inside the Debian container, with /build mounted:
apt-get install -y kpartx
losetup -fP --show dm250-stock.img    # e.g. /dev/loop0
ls /dev/loop0p*                       # see partitions
mkdir -p /mnt/rootfs

# Try each partition; the rootfs is usually the largest ext4/squashfs.
mount -o ro /dev/loop0pN /mnt/rootfs
ls /mnt/rootfs/lib/firmware/brcm/
mkdir -p /build/firmware-backup
cp -a /mnt/rootfs/lib/firmware/brcm /build/firmware-backup/
cp -a /mnt/rootfs/etc/firmware /build/firmware-backup/ 2>/dev/null || true
umount /mnt/rootfs
losetup -d /dev/loop0

If the rootfs is squashfs (likely — common for read-only embedded), use unsquashfs on the partition image instead of mount.

Filenames matter. brcmfmac matches firmware by a name derived from chip + revision + the brcm,bcm4329-fmac compatible string in the DT. Don't rename files when copying. Copy the entire brcm/ directory verbatim into your new rootfs at /lib/firmware/brcm/.

Verifying the backup before doing anything risky

  • file dm250-stock.img → should report a partitioned disk image.
  • fdisk -l dm250-stock.img (or gdisk -l) → partition table visible.
  • strings dm250-stock.img | grep -i pomera → should find references in u-boot and/or kernel command line, confirming the dump captured the bootloader region.
  • ls firmware-backup/brcm/ → should show at least .bin, .txt, and .clm_blob files for whatever 43xxx chip variant your unit has.

Restoring (if something goes wrong later)

# Boot DM250 back into MASKROM, push the loader as before, then:
rkdeveloptool wl 0 dm250-stock.img    # Write Linear — restores byte-for-byte
rkdeveloptool rd

This makes any later experiment recoverable, including a totally bricked eMMC bootloader. The MASKROM stage lives in on-die ROM and cannot be broken by software.

Caveat: If KING JIM blew the SECURE_BOOT eFuse on the RK3128 (signed boot), MASKROM mode may still work for reading, but loader push may fail because the chip refuses to execute unsigned loaders. The community consensus is that consumer Pomera units are not fused, but verify with rkdeveloptool db — if it succeeds, you're fine.

Choosing the fork base

For a plain DM250 (non-US), the jcs fork is the better starting point than the asciilifeform base:

  • 17 commits ahead, 0 behind asciilifeform. Strict superset for DM250 hardware.
  • Fixes real polarity / inversion bugs (tc35894 reset, SDMMC CD).
  • Fleshes out Wi-Fi/SDIO + Bluetooth GPIOs.
  • Delta is device-tree only — trivial to rebase onto newer mainline.

Dev environment setup

Tooling decision: Docker vs. Homebrew vs. VM

OptionProsCons
Docker (recommended) Reproducible. Uses Debian/Ubuntu's gcc-arm-linux-gnueabihf. No host pollution. Works identically on Intel and Apple Silicon. ~1.5× slower than native because the container is linux/arm64 running a 32-bit cross-compiler. Still fast on M-series.
Homebrew cross toolchain Native, fastest. No first-class formula. Needs the messense/macos-cross-toolchains tap. Case-insensitive FS, BSD sed, missing elf.h all bite.
Lima / UTM Linux VM Full Linux build host, no Docker layer. Disk/RAM overhead. More setup than Docker.

Plan below uses Docker.

Docker setup on macOS

Install runtime

# OrbStack is faster than Docker Desktop on Apple Silicon
brew install --cask orbstack
# OR
brew install --cask docker
Known gotcha: if you previously had Docker Desktop and uninstalled it, ~/.docker/config.json may still contain "credsStore": "desktop". Any docker pull then fails with docker-credential-desktop: executable file not found in $PATH. Public images don't need credentials — just strip the field: 
jq 'del(.credsStore)' ~/.docker/config.json > ~/.docker/config.json.new \
  && mv ~/.docker/config.json.new ~/.docker/config.json
Also confirm a working context: docker context ls, then docker context use orbstack (or default) if desktop-linux is stale.

Launch the builder container

cd ~/src/linux-dm250

docker run --rm -it \
  --platform=linux/arm64 \
  -v "$PWD":/build -w /build \
  debian:trixie bash

Install the toolchain inside the container

apt-get update
apt-get install -y --no-install-recommends \
  build-essential bc bison flex libssl-dev libelf-dev \
  cpio kmod rsync git \
  gcc-arm-linux-gnueabihf
arm-linux-gnueabihf-gcc --version    # GCC 13/14 expected
One-time builder image: a ready-made Dockerfile at ~/src/dm250-tools/Dockerfile bakes in all the kernel-build and SD-assembly packages. Build once with: 
docker build --platform=linux/arm64 -t dm250-builder ~/src/dm250-tools
Then use dm250-builder as the image in docker run everywhere below.

GCC version

Mainline kernel requires GCC ≥ 8.1 per Documentation/process/changes.rst. Newer is fine.

GCCRecommendation
8.1 – 12Works.
13 – 14Recommended. Debian trixie / Ubuntu 24.04 default.
15 (latest)Should work for v6.16. Expect occasional -Wmaybe-uninitialized noise.

Simplest choice: whatever gcc-arm-linux-gnueabihf Debian/Ubuntu ships. Don't fight the toolchain.

Why not Clang? Kernel supports Clang/LLVM, but RK3128 ARMv7 board builds are GCC-tested in practice. Stick with GCC unless you have a reason.

Verify .config in one shot

grep -E "^CONFIG_(ARCH_ROCKCHIP|MFD_RK8XX_I2C|REGULATOR_RK808|MFD_TC3589X|KEYBOARD_TC3589X|GPIO_TC3589X|DRM_PANEL_LVDS|ROCKCHIP_LVDS|DRM_ROCKCHIP|ROCKCHIP_VOP|BACKLIGHT_PWM|PWM_ROCKCHIP|ROCKCHIP_SARADC|KEYBOARD_GPIO|LEDS_GPIO|REGULATOR_FIXED_VOLTAGE|PWRSEQ_SIMPLE|MMC_DW|MMC_DW_ROCKCHIP|BRCMFMAC|BT_HCIUART_BCM|RTC_DRV_RK808|ROCKCHIP_THERMAL|DW_WATCHDOG|I2C_RK3X|DRM_LIMA)=" .config | sort

Anything missing from the output is unset — turn it on via make menuconfig.

Once a known-good .config boots end-to-end on hardware, run make savedefconfig and commit the result as arch/arm/configs/pomera_defconfig. Future rebuilds reduce to one command.

=y vs =m rule of thumb

Anything you need in the path from power-on to "shell prompt" → =y. Everything else → =m to keep zImage small.

  • Must be y: MMC_DW, MMC_DW_ROCKCHIP, rootfs FS, PMIC, clock/pinctrl/regulator core, MFD_TC3589X (it's bool — no choice).
  • Fine as m: Wi-Fi, BT, Lima GPU, charger driver, hwmon. Loaded once userspace is up.
  • Keyboard: =y if it's your only input; =m if you have serial console.

Build & flash

Build commands

NPROC=$(nproc)
make -j"$NPROC" zImage modules dtbs

Outputs:

  • arch/arm/boot/zImage — the kernel image
  • arch/arm/boot/dts/rockchip/pomera-dm250.dtb — the device tree blob
  • ./modules/ after make modules_install INSTALL_MOD_PATH=./modules

Sanity checks before flashing

  • file arch/arm/boot/zImage → Linux kernel ARM boot executable zImage (little-endian)
  • strings arch/arm/boot/dts/rockchip/pomera-dm250.dtb | grep -i pomera → should show KING JIM Pomera DM250 (jcs fork) or Rockchip RK3128 Pomera DM250 (asciilifeform).
  • grep -E "ROCKCHIP|LIMA|TC3589" .config → rockchip drivers + TC3589X enabled.
  • Confirm firmware blobs from the backup step are staged into the new rootfs at /lib/firmware/brcm/.

Producing a bootable image

For RK3128 + u-boot the usual recipe is a FIT image or a uImage wrapper. If your boot pipeline expects zImage + .dtb side-by-side, stop after the build step.

Recommended first deployment: SD card, not eMMC. See SD-first validation for a non-destructive way to confirm the kernel boots and drives the hardware before touching internal storage.

# u-boot legacy uImage (only if your bootloader wants it)
make -j"$NPROC" UIMAGE_LOADADDR=0x60008000 uImage

SD-first validation

Why boot from SD first

Two-step shortcut available. A companion script at ~/src/dm250-tools/mk-sd.sh automates everything in this section — u-boot build, idbloader assembly, Alpine rootfs, partitioning, loader-sector writes. Usage: 
docker run --rm --privileged \
  -v "$HOME/src/linux-dm250":/build/linux-dm250 \
  -v "$HOME/src/dm250-tools":/build/tools \
  -w /build dm250-builder \
  /build/tools/mk-sd.sh /build/sdcard.img

sudo dd if=~/src/linux-dm250/sdcard.img of=/dev/rdiskN bs=4m status=progress
See ~/src/dm250-tools/README.md for env-var knobs (firmware dir, image size, u-boot tag). The rest of this section explains what the script is doing under the hood — useful if you want to customize or debug.

The RK3128 BootROM checks the SD card before eMMC for a valid loader (idbloader.img at sector 64). If you stage a complete bootable image on SD, you can validate your kernel, DTB, and driver coverage without overwriting the stock eMMC. Pull the SD card and the device boots stock again.

The fork's pomera-dm250.dts already assumes this workflow:

aliases {
    mmc0 = &emmc;     /* internal eMMC */
    mmc1 = &sdmmc;    /* SD card slot  */
};
chosen {
    bootargs = "... root=/dev/mmcblk1p2 rw rootwait";
};

mmcblk1p2 is the second partition of the SD card. asciilifeform designed the port for SD-first development; jcs's fork keeps it.

SD card layout

RK3128 expects the loader at fixed sector offsets, before the partition table's data area. Standard Rockchip layout:

Sector offsetSizeContents
01 sectorGPT protective MBR + GPT header
64~4 MBidbloader.img (DDR init + SPL, built from rkbin)
16384~4 MBu-boot.itb (FIT image with u-boot + ATF/OP-TEE if used)
partition 1~256 MB/boot (ext4 or FAT) with zImage, pomera-dm250.dtb, extlinux/extlinux.conf
partition 2rest/ (ext4) — rootfs with /lib/firmware/brcm/ blobs from backup step

Minimal extlinux/extlinux.conf:

default linux-dm250
prompt 0
timeout 1

label linux-dm250
    kernel /zImage
    fdt /pomera-dm250.dtb
    append earlycon=uart8250,mmio32,0x20064000 console=ttyS1,115200n8 root=/dev/mmcblk1p2 rw rootwait
Building idbloader.img and u-boot.itb is a separate exercise from kernel build. Easiest path: clone u-boot mainline, make rk3128_defconfig && make CROSS_COMPILE=arm-linux-gnueabihf-, then run tools/mkimage with the appropriate Rockchip DDR/miniloader blobs from rkbin. Both repos live on GitHub. Pomera-specific DT support in mainline u-boot is sparse — you may need to add a minimal u-boot DTS for RK3128 board variant. Or scavenge u-boot from asciilifeform's notes if published.

Tool for writing the loader at sector 64 from macOS:

# From inside the build container (SD card visible as /dev/disk?):
dd if=idbloader.img of=/dev/sdX seek=64 conv=notrunc
dd if=u-boot.itb   of=/dev/sdX seek=16384 conv=notrunc
sync

On macOS host (outside Docker), the device is /dev/rdiskN. Unmount first with diskutil unmountDisk /dev/diskN. Triple-check the device name — dd to the wrong disk overwrites your Mac's SSD.

Boot & validate workflow

  1. Build kernel + DTB + modules per the previous sections.
  2. Build u-boot for RK3128 (or reuse a prebuilt one if available).
  3. Partition SD card; dd idbloader + u-boot.itb at fixed offsets; format and populate partitions.
  4. Install modules_install INSTALL_MOD_PATH=<sd-mountpoint>.
  5. Copy the backed-up brcmfmac / BT blobs into <rootfs>/lib/firmware/brcm/.
  6. Attach a serial console to uart1 (115200 8N1). Without this, a failed boot is silent.
  7. Insert SD, power on. Watch boot messages.
  8. If stock u-boot runs first and ignores SD: interrupt at the u-boot prompt over serial, then manually: 
    # In u-boot:
mmc dev 1                       # switch to SD
load mmc 1:1 0x62000000 zImage
load mmc 1:1 0x68000000 pomera-dm250.dtb
setenv bootargs 'console=ttyS1,115200n8 root=/dev/mmcblk1p2 rw rootwait'
bootz 0x62000000 - 0x68000000
    This confirms the SD boot path before you commit to eMMC reflash.
  9. Once Linux comes up, validate hardware:
    • Keyboard: evtest /dev/input/event*
    • LCD: kernel should bring up the panel and print fbcon messages
    • Wi-Fi: dmesg | grep brcmfmac, then ip link set wlan0 up
    • SD/MMC: lsblk
    • Battery: will not appear — see RK818 gap
  10. Only after the SD-booted kernel passes validation, decide whether to flash eMMC.

Reversibility caveats

  • Stock u-boot on eMMC still runs first. If it's locked down / signed and refuses to chain-load from SD, you may need BootROM mode (boot-key combo at power-on, or shorting BOOT pin) to force SD-only boot. Confirm with asciilifeform's writeup before assuming behavior.
  • u-boot env lives in eMMC. Any saveenv at the u-boot prompt mutates eMMC. To stay non-destructive, never saveenv — type bootcmds at the prompt each session, or set env in memory only.
  • BootROM SD-first behavior is not absolute on all Rockchip designs; some boards have BOOT_SEL fused to eMMC. If your DM250 stubbornly ignores the SD card even from a cold boot with no eMMC chainload, BOOT_SEL is the suspect.
  • The serial console is mandatory for this workflow. Locate the UART pads on the DM250 PCB and solder leads before powering on. Otherwise a failed boot tells you nothing.

Known driver gaps

RK818 charger/battery have no mainline Linux driver

The device tree includes rockchip,rk818-charger and rk818-battery nodes, and the fork has a commit titled "rk818-charger and rk818-battery work on OpenBSD" which re-enables them. That title is precise:

  • OpenBSD has drivers for the RK818 charger and fuel-gauge sub-blocks.
  • Linux mainline does not. A grep through drivers/ shows no driver bound to either compatible string. The PMIC core (MFD_RK8XX_I2C) provides regulators and RTC only.

Practical effect on a Linux DM250:

  • Power rails work (regulators bring up CPU/RAM/peripherals).
  • RTC works.
  • Battery percentage, charge state, and charging current are not reported. Userspace (upower etc.) sees no battery.
  • Charging itself still happens in hardware — the RK818 PMIC's charger is autonomous — but Linux can't query or configure it.

Options:

  1. Live without battery reporting (it still charges).
  2. Port the OpenBSD rk818_charger driver to Linux. Non-trivial but bounded — the chip is documented in the RK818 datasheet, and OpenBSD's implementation is small.
  3. Watch upstream: there have been periodic patches on the linux-rockchip list adding RK817/RK818 charger support, none merged as of v6.16-rc1.

Don't waste time hunting for a Kconfig — it doesn't exist yet.

brcmfmac requires firmware blobs from stock

The BRCMFMAC driver alone is useless. It loads but the radio refuses to come up without:

  • brcmfmac43430-sdio.bin — the firmware itself
  • brcmfmac43430-sdio.<board>.txt — NVRAM with per-board calibration + MAC address
  • brcmfmac43430-sdio.clm_blob — channel/locale matrix
  • BCM43430A1.hcd — BT patchram for the BCM43438 BT side

Exact filenames depend on the chip revision the driver detects. None of these are redistributable; mainline linux-firmware ships some 43xx blobs but not all variants and rarely the per-board NVRAM. You must extract them from the stock partition. See back up stock firmware.
